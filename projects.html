OwnAI

Self-Hosted Local AI Server

Personal / Homelab

In Development
▼

Overview

A fully self-hosted AI assistant running on local hardware — no cloud APIs, no data leaving the machine. Built on a Supermicro server with an RTX 2060, it runs Ollama for LLM inference and AnythingLLM as the chat interface with RAG (Retrieval-Augmented Generation) over private documentation. The current knowledge base covers the full Proxmox VE Administration Guide and LME infrastructure notes.

Next step: integrate a messenger bot (Signal / Telegram via n8n) for a personal assistant that handles reminders, notes, and weekly planning — with LLM as the language frontend to fixed deterministic logic.

Technology Stack

🖥️

Supermicro + RTX 2060

Intel Xeon 12-core, 16 GB DDR4 ECC, NVIDIA RTX 2060 6 GB VRAM

🦙

Ollama

LLM runtime — runs natively (not containerised) for direct GPU access. Models: llama3.1:8b + qwen2.5-coder:7b

💬

AnythingLLM

Chat UI and RAG pipeline in Docker — LanceDB vector store, document ingestion, workspace management

🐧

Ubuntu Server 22.04

Host OS with NVIDIA drivers, systemd service management

🐳

Docker

Container isolation for AnythingLLM; Ollama runs native to avoid NVIDIA Container Toolkit overhead

Key Features

01

100% Local — No Cloud

All inference runs on the RTX 2060. No API keys, no data leaving the server, full data sovereignty

02

RAG over Private Docs

Proxmox VE Administration Guide (685 pages, split into 21 chapters) ingested as a knowledge base. RAG returns contextually relevant answers in German and English

03

GPU Optimisation

KV-cache quantisation (q8_0) + Flash Attention keeps llama3.1:8b at 100% GPU with 8192 context — no CPU spill

04

Multi-Model Workspaces

llama3.1:8b for general Q&A in German/English; qwen2.5-coder:7b for code and infrastructure tasks

What I Learned

  • GPU resource management — VRAM budgeting, KV-cache quantisation, context-window tuning to keep the model 100% on GPU
  • Docker networking — bridging containers to natively-running host services via host.docker.internal
  • RAG pipeline design — chunking strategy, PDF-to-Markdown pre-processing, embedder vs. LLM VRAM trade-offs
  • Systemd unit overrides — exposing Ollama on 0.0.0.0 instead of loopback so Docker can reach it
  • Model selection — a code-specialised model (qwen2.5-coder) is not a stronger general model, it is a different model
  • Methodical root-cause debugging — reading container logs to trace an OOM kill back to chunk size, not model size
